# CR 快速打开与定义跳转可靠性设计

## 目标

修复两个代码阅读问题：

1. `Command + P` 快速打开列表显示为浏览器原生按钮堆叠，缺少清晰的面板层级、列表状态和滚动边界。
2. 工作区代码在实际默认浏览器中无法稳定通过 `Command + 鼠标左键` 跳转到定义。

保留 `F12` 作为与鼠标事件无关的稳定跳转方式，并在没有定义时给出明确反馈。

## 已确认根因与边界

### 快速打开

`QuickOpen` 已有搜索和打开文件逻辑，但 `.quick-open` 只覆盖弹层容器与输入框。结果列表仍继承浏览器默认的 `ul`、`li` 和 `button` 外观，没有最大高度、滚动、整行点击区域、悬停态、焦点态或长路径省略，因此在大工程中表现为截图中的原生按钮堆叠。

### 定义跳转

真实 `tab-commerce` 工程的 Python 定义接口可以把 `member_controller.py` 中的 `_service.get_level` 正确解析到 `member_service.py`。标准 Chrome 中，当前自定义 Monaco `onMouseDown` 监听也能发出定义请求并打开目标文件。因此服务端解析、文件读取和工作区导航不是本次根因。

CR 通过系统默认浏览器打开。现有监听只依赖 Monaco 包装后的 `metaKey` 和 `leftButton` 字段，且没有 Monaco 原生的可跳转符号反馈。浏览器事件表示或命中反馈出现差异时，用户无法判断符号是否可跳转，也没有失败提示。

本次不修改服务端语言导航，不扩大 Python/TypeScript 的可解析语义范围，不新增 Peek Definition，也不改变现有导航历史快捷键。

## 方案

采用方案 A：Monaco 原生定义反馈、显式点击兼容和 `F12` 兜底并存。

### Command+P 命令面板

保持 `QuickOpen` 的搜索数据流和最多 20 条结果不变，补全紧凑命令面板样式：

- 面板固定在阅读区上方居中，具有明确边框、圆角、阴影和遮罩层级。
- 输入框使用现有 CR 主题色，焦点状态与左侧文件筛选框一致。
- 结果区设置最大高度并独立滚动，避免大工程列表撑满页面。
- 每个结果是无原生边框的整行按钮；悬停和键盘焦点使用柔和底色。
- 长路径单行省略，同时通过按钮标题保留完整路径。
- 关闭按钮使用次要操作样式，不再显示为浏览器默认按钮。
- 亮色、暗色以及窄窗口继续使用现有设计 token，不引入硬编码的第二套主题。

本次只修复视觉与可读性，不增加结果排序、模糊匹配或键盘上下选择等新功能。

### 无副作用的 Monaco Definition Provider

重新注册 Python 和 TypeScript Definition Provider，但 Provider 只执行查询并返回 Monaco 定义位置，不直接调用工作区 `onNavigate`。

Command 悬停时，Monaco 可以查询定义并显示原生可点击符号反馈；由于 Provider 不包含导航副作用，悬停不会切换文件。目标位置使用 CR 自有 URI 表示，注册的 editor opener 只接管当前 CR 编辑器发出的该类 URI，并把它转换回现有 `SourceLocation` 后调用 `onNavigate`。

Provider、editor opener 和鼠标监听都必须在编辑器重挂载或组件卸载时释放，避免重复请求与陈旧闭包。

### 显式 Command+左键兼容

保留显式鼠标监听作为浏览器兼容路径。一次点击必须同时满足：

- Command 已按下：优先读取 Monaco `metaKey`，并用原始 `browserEvent.metaKey` 兜底。
- 鼠标左键：接受 Monaco `leftButton`，并用原始 `browserEvent.button === 0` 兜底。
- 点击目标包含有效 Monaco 代码位置。

显式路径复用同一个定义解析函数。若原生 Provider 已查询相同位置，应复用进行中的请求或短期结果，避免一次 Command+点击产生重复的服务端请求和重复导航。显式路径处理后阻止同一次鼠标事件继续触发第二次原生打开。

普通点击、右键、中键、仅 Command 悬停或没有代码位置的点击均不导航。

### F12 与失败提示

`F12` 继续读取当前光标位置并直接调用同一个定义解析函数，不依赖鼠标事件或浏览器兼容字段。

显式 Command+点击或 `F12` 没有解析到目标时，在编辑器区域显示约 2 秒的非阻塞状态提示：

> 未找到定义，可将光标置于符号上按 F12 重试

Provider 为悬停反馈进行的预查询若没有结果，不显示提示，避免移动鼠标时反复打扰用户。接口请求失败时显示“定义跳转失败，请按 F12 重试”，并保持当前文件和导航历史不变。

提示使用 `role="status"` 和 `aria-live="polite"`，不会抢走 Monaco 焦点。

## 数据流

1. Command 悬停符号时，Provider 请求现有 `/api/navigation/definition`。
2. 有定义时，Provider 返回目标 URI 与位置，Monaco 显示原生链接反馈；无定义时安静返回空。
3. Command+左键点击时，兼容监听确认修饰键、左键和代码位置，复用缓存的解析结果或发起查询。
4. 有目标时调用现有 `onNavigate`，由 Zustand 工作区写入导航历史并打开目标文件。
5. 无目标或请求失败时显示短暂提示，不改变当前文件。
6. `F12` 从当前光标位置直接进入第 3 步的显式解析路径。

## 测试与验收

### 组件测试

- Command+P 面板结果按钮不再使用未分类的原生结构，并具有可测试的面板、结果区和关闭操作类名。
- 长结果列表由结果区滚动承载，结果仍限制为 20 条。
- Provider 查询定义时不会直接导航。
- 原生 opener 可以把 CR URI 转换为工作区位置并导航。
- Monaco 包装字段与原始浏览器字段两种 Command+左键事件都能跳转。
- 普通点击、非左键和无位置点击不会请求定义。
- 相同位置的 Provider 与显式点击不会产生重复导航。
- `F12` 可以独立跳转。
- 无定义和请求失败显示对应提示，悬停预查询不显示提示。
- 所有 disposable 和提示定时器在卸载时清理。

### 真实浏览器与打包验收

- Command+P 在大工程中显示为紧凑命令面板，结果可滚动，长路径不会撑破面板。
- Command 悬停可解析符号时出现 Monaco 原生反馈，但文件不切换。
- Command+左键点击可解析的 Python 或 TypeScript 符号时打开定义。
- 无定义时显示约 2 秒提示，当前文件保持不变。
- 将光标放在可解析符号上按 `F12` 可以跳转。
- `Command + -` 和 `Command + Shift + -` 仍可后退和前进。
- 完整测试、类型检查、构建、E2E、Swift launcher、macOS bundle、签名和 smoke 验证通过后，才替换共享 `outputs/CR.app`。

如果系统默认浏览器无法由自动化工具直接接管，则以打包后的默认浏览器手工验收步骤和自动化 Chromium 回归共同作为交付证据，并明确记录该限制。
