# CR 工程路径与隐藏文件过滤设计

## 目标

在不改变现有本地工程读取和安全边界的前提下：

1. 在顶部工具栏展示当前工程的绝对路径。
2. 在左侧文件区域提供“显示隐藏文件”开关。
3. 让该开关同时控制文件树与 `⌘P` 文件搜索结果。

## 交互设计

### 工程绝对路径

- 当前工程名称后展示 `ProjectSummary.root`。
- 路径使用弱化颜色与等宽字体，避免抢占工程名称和主要操作的视觉层级。
- 工具栏空间不足时路径使用单行省略号截断。
- 路径元素的 `title` 保存完整值，悬停时可查看完整绝对路径。
- 未打开工程时不展示路径。

### 隐藏文件开关

- 在左侧“搜索文件”入口下方增加复选开关，标签为“显示隐藏文件”。
- 开关默认关闭，并且本次不持久化；刷新页面或重新启动 CR 后恢复为关闭。
- 名称以 `.` 开头的文件或目录属于隐藏项。
- 关闭时，从文件树和 `⌘P` 搜索候选中递归移除隐藏项。
- 打开时，显示服务端文件树中存在的隐藏项。
- 工程根目录不是树节点，不参与隐藏项判断。

## 永久忽略规则

现有服务端继续永久忽略 `.git`、`node_modules`、虚拟环境、缓存和构建产物等目录。前端开关不改变这些规则，也不新增后端参数。这样可以避免读取大型生成目录并保持现有性能与安全边界。

## 架构与数据流

采用前端统一过滤方案：

1. `FileBrowser` 继续通过现有 `/api/project/tree` 获取文件树。
2. 新增纯函数，递归过滤名称以 `.` 开头的节点，并保持其余节点的层级和顺序。
3. `FileBrowser` 根据开关状态计算唯一的可见树。
4. 可见树同时传给 `QuickOpen` 和文件树渲染组件，确保两处结果一致。
5. 切换仅改变前端派生数据，不重新请求服务端，也不触发重新索引。

目录在移除隐藏子项后即使为空也继续展示，因为目录自身仍是可见的工程结构。

## 组件边界

- `ProjectToolbar`：负责展示工程名称、绝对路径和现有打开工程操作。
- `FileBrowser`：持有“显示隐藏文件”的页面内状态，并生成可见树。
- 文件树过滤纯函数：只负责根据节点名称生成过滤后的新树，不修改原始查询结果。
- `QuickOpen`：保持现有职责，只接收已经过滤的树。

不修改服务端、共享契约或工程设置存储。

## 错误处理

- 路径来自已验证的当前工程摘要，不增加额外请求或新错误状态。
- 文件树请求失败时继续使用现有“无法读取工程文件”提示。
- 切换过滤不执行 I/O，因此不增加加载或失败状态。

## 测试

- `ProjectToolbar` 组件测试验证已打开工程时展示完整绝对路径，并带有完整路径提示。
- `FileBrowser` 组件测试验证默认隐藏点文件和点目录。
- `FileBrowser` 组件测试验证打开开关后隐藏项出现。
- 搜索交互测试验证关闭时搜索不到隐藏文件，打开后能够搜索并打开隐藏文件。
- 运行现有 Web 测试、全仓测试、类型检查和构建，确认没有回归。

## 验收标准

- 打开工程后，顶部可识别当前工程的绝对路径，长路径不会挤坏工具栏。
- 左侧存在清晰的“显示隐藏文件”开关，默认关闭。
- 点文件和点目录在文件树及 `⌘P` 搜索中同步隐藏或显示。
- 永久忽略目录不会因开关恢复。
- 现有工程打开、文件跳转、Controller 和 Enum 功能保持不变。
