# CR Dock 常驻与服务生命周期设计规格

## 1. 目标

让用户启动 `CR.app` 后能在 macOS Dock 中持续看到 CR 图标。CR 仍以浏览器作为主界面，但原生启动器不再在打开浏览器后立即退出；用户点击正在运行的 Dock 图标时可以重新打开 CR 页面，退出 CR 时停止对应的本地服务。

## 2. 已确认行为

采用用户确认的常驻方案：

- 首次打开 `CR.app`：显示正在运行的 Dock 图标，启动或复用本地 CR 服务，并在默认浏览器打开页面。
- 浏览器页面关闭后：CR 原生进程继续运行，Dock 图标继续显示。
- 点击正在运行的 CR Dock 图标：校验服务状态；服务可用时重新打开页面，服务已因空闲等原因退出时先重新启动再打开页面。
- 通过 Dock 菜单“退出”、应用菜单“退出 CR”或 `Cmd+Q` 退出：停止当前 CR 服务、清理启动状态文件并结束原生进程，Dock 运行指示随之消失。
- 用户可以像普通 macOS App 一样把 CR 图标固定到 Dock。固定后的图标在 CR 退出后仍作为启动快捷方式保留，这是 macOS Dock 的用户级行为，不由 CR 自动修改。

默认浏览器行为保持为系统级选择：CR 继续通过 `NSWorkspace.shared.open(URL)` 打开本地页面，不写死 Safari、Chrome 或其他浏览器的 Bundle Identifier，也不修改用户的默认浏览器设置。

## 3. 已确认根因

当前产物设置 `LSUIElement=true`，macOS 因此把 CR 作为后台辅助应用处理并隐藏其 Dock 图标。当前 `CRLauncherMain.main()` 在 `ServiceLauncher.launchOrReuse()` 打开浏览器后返回，原生启动器随即退出，只留下无 Dock 图标的 Node.js 本地服务。

图标资源本身不是故障点：产物已经声明 `CFBundleIconFile=CR`，且 `Contents/Resources/CR.icns` 存在并通过包测试。

## 4. 原生应用生命周期

### 4.1 应用启动

启动入口创建 `NSApplication`，设置普通应用激活策略并安装应用代理，然后进入 AppKit 事件循环。`Info.plist` 删除 `LSUIElement`，使系统按普通前台应用展示 Dock 图标。

应用代理在 `applicationDidFinishLaunching` 中异步调用现有服务启动逻辑。服务就绪并打开浏览器后，原生应用保持运行，但不创建业务窗口；Dock 图标是原生侧唯一持续可见的界面入口。

### 4.2 Dock 重新打开

应用没有可见窗口时，macOS 对运行中 Dock 图标的点击通过重新打开回调交给应用代理。回调触发一次受控的“启动或复用并打开浏览器”操作：

1. 读取本地 `service.json`。
2. 继续使用现有的 UID、可执行文件路径、PID 和健康检查校验。
3. 校验成功则打开当前服务 URL。
4. 校验失败则清理过期状态，启动新服务，等待健康检查通过后打开 URL。

同一时刻只允许一个打开任务运行，避免连续点击 Dock 图标并发启动多个服务。

### 4.3 退出

应用代理在终止回调中调用同步、幂等的停止逻辑。停止逻辑只对满足以下全部条件的进程发送终止信号：

- 状态记录的 UID 是当前用户。
- PID 当前存在。
- PID 对应的实际可执行文件与 App 包内 Node.js 的规范路径一致。

校验失败、进程已经退出或状态文件损坏时，不终止其他进程，只删除无效的 CR 状态文件。正常停止后同样删除状态文件。该规则防止 PID 复用导致误杀无关进程。

## 5. 菜单与交互

CR 不新增原生业务窗口。启动器创建最小应用菜单，包含系统标准的“关于 CR”“隐藏 CR”和“退出 CR”；`Cmd+Q` 使用标准退出动作。

启动失败时继续显示现有中文错误弹窗。用户确认弹窗后应用退出，避免留下一个无法打开页面但仍显示为运行中的 Dock 图标。Dock 重新打开失败时显示同一类错误弹窗，但保留应用进程，用户可再次点击重试或主动退出。

## 6. 代码边界

- `CRLauncherMain.swift`：只负责创建 `NSApplication`、安装代理、配置激活策略和运行事件循环。
- 新增 `CRApplicationDelegate.swift`：负责启动、Dock 重新打开、并发打开保护、错误呈现和退出回调。
- `ServiceLauncher.swift`：继续负责服务资源校验、启动、复用和打开浏览器；新增经过进程身份校验的幂等停止能力。
- `ServiceState.swift`：复用现有状态与进程身份验证，不降低任何校验条件。
- `Info.plist.in`：删除 `LSUIElement`，保留已经接入的 `CFBundleIconFile=CR`。
- macOS 包测试：同时断言 `LSUIElement` 不存在、图标声明正确、图标资源存在且签名有效。

## 7. 路径与品牌清理边界

仓库内的产品源码、测试、脚本、配置、示例工程和交付文档不得包含以下内容：

- 开发者机器的用户主目录或系统临时输入路径、个人工作区目录、本次 Codex 工作树路径。
- 不属于 CR 的旧产品命名。
- 为特定浏览器写死的应用路径或 Bundle Identifier。

构建和交付脚本使用脚本自身位置推导的 `repo_root`，文档中的命令使用项目相对路径或 `${repo_root}`。现有图标实施计划里的 4 处开发机绝对路径改为 `outputs/CR.app` 或由 `repo_root` 推导的路径。

以下内容不属于需要替换的本地路径：

- Controller 示例中的 `/users/:id`、`/users/{user_id}` 等 HTTP 路径。
- 测试使用的通用虚拟路径，例如 `/work/mixed-project`。
- 用户运行 CR 时主动选择的工程绝对路径。该路径是重新打开最近工程所需的用户数据，不是代码中写死的开发机路径，继续保存在用户本地设置中。
- Git worktree 的 `.git` 指针。它是 Git 自动生成的仓库元数据，不进入产品源码或 App 交付内容。

在验收阶段对全部 Git 跟踪文件执行大小写不敏感的旧品牌扫描，以及大小写敏感的开发者主目录、临时输入目录、个人目录和工作树路径扫描。任何产品文件命中都必须在交付前清理；HTTP Controller 路径不应被本地路径规则误判。

## 8. 错误处理

- 启动或重开操作异常不能阻塞 AppKit 主线程。
- 连续点击 Dock 时忽略正在执行期间的重复请求；当前请求结束后允许再次尝试。
- 服务状态文件损坏或过期时按现有逻辑清理并重建。
- 退出时停止失败不阻止原生应用退出，但绝不绕过 UID 和可执行文件校验去强制终止未知进程。
- 测试模式沿用 `CR_TEST_MODE` 和独立支持目录，不能操作用户真实的 CR 服务状态。

## 9. 测试与验收

自动化测试覆盖：

1. 包测试先证明当前 `LSUIElement=true` 不符合 Dock 可见契约，再验证新包中该键不存在。
2. 服务停止测试验证：匹配当前 UID 和 Node 路径时发送终止信号并删除状态；UID 不匹配、可执行文件不匹配、PID 不存在和损坏状态时不终止其他进程。
3. 应用代理测试验证：首次启动调用一次打开逻辑；打开任务进行中时重复 Dock 点击不并发调用；任务完成后可以再次调用。
4. 现有 launcher、Node 服务、浏览器 UI 和包测试继续通过。
5. 全部 Git 跟踪文件不包含开发机主目录、临时输入路径、个人工作区路径或旧产品命名；默认浏览器调用仍为 `NSWorkspace.shared.open`。

手工/集成验收覆盖：

1. 双击共享交付目录中的 `CR.app` 后，Dock 出现带新 CR 图标的运行项，浏览器打开 CR。
2. 关闭浏览器标签页后 Dock 图标仍在；再次点击图标会重新打开 CR。
3. 服务因空闲退出后再次点击图标，CR 能重新启动服务并打开页面。
4. 从 Dock 退出 CR 后，原生进程和对应本地服务均结束，状态文件被清理。
5. `npm test`、类型检查、前后端构建、arm64 Swift release 构建、macOS 包测试、启动 smoke 和严格签名验证全部通过。

## 10. 范围外事项

本次不把浏览器 UI 改写为原生窗口，不自动把 CR 固定到用户 Dock，不增加菜单栏常驻项，不改变服务的 15 分钟浏览器空闲退出策略，也不改变 CR 的业务功能、图标设计、Bundle Identifier 或版本号。
