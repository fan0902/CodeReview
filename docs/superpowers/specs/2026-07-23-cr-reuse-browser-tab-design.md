# CR 复用浏览器标签页设计规格

## 1. 目标

用户首次启动 `CR.app` 时，CR 继续使用 macOS 系统默认浏览器打开页面。CR 本地服务仍在运行且浏览器中已有 CR 标签页时，再次点击 CR App 不创建新标签页，而是在原标签页中刷新页面。

如果同时存在多个 CR 标签页，只刷新最近活跃的一个。没有可用 CR 标签页、页面连接已经失效或本地服务需要重新启动时，继续通过系统默认浏览器新开 CR 页面，保证用户始终能够进入应用。

## 2. 已确认行为

- 首次启动 CR：启动本地服务，并在系统默认浏览器新开一个 CR 标签页。
- 已有一个 CR 标签页：再次点击 CR App，原标签页执行完整页面刷新，不增加标签页。
- 已有多个 CR 标签页：只刷新最近活跃的标签页，其他 CR 标签页保持不变。
- CR 标签页均已关闭或连接失效：再次点击 CR App，新开一个 CR 标签页。
- 本地服务不可复用：清理失效状态、启动新服务并新开 CR 标签页。
- 刷新已有页面时，CR 尝试把系统默认浏览器激活到前台，但不通过打开 URL 的方式激活，因此不会因此创建额外标签页。

“最近活跃”按页面发给本地服务的活动心跳顺序确定。页面首次加载、重新变为可见以及用户在页面中进行指针或键盘操作时都会更新顺序；时间相同时仍以服务端接收心跳的先后顺序为准。

## 3. 根因

当前 `ServiceLauncher.launchOrReuse()` 在两条成功路径中都会执行 `NSWorkspace.shared.open(state.launchURL)`：

1. 复用现有本地服务时直接打开 URL。
2. 启动新服务并通过健康检查后打开 URL。

`NSWorkspace.shared.open` 会把 HTTP URL 交给默认浏览器，但不能表达“刷新已经存在的标签页”。因此每次点击运行中的 CR Dock 图标都会再次打开 URL，并由浏览器创建新标签页。应用代理中的并发门只会阻止同一时刻的重复请求，不能阻止先后两次点击各自新建标签页。

## 4. 方案选择

采用“本地服务推送页面指令”方案：每个 CR 页面与本地服务建立一个经过现有会话认证的轻量事件连接；启动器请求本地服务刷新最近活跃页面，服务把刷新指令只推送给被选中的连接。

未采用以下方案：

- 页面定时轮询：能够保持浏览器无关，但每个页面会持续产生短周期请求，刷新也存在轮询间隔带来的延迟。
- AppleScript 或浏览器扩展查找标签：可以直接操控标签页，但会绑定 Safari、Chrome 或其他具体浏览器，违背系统默认浏览器约束，也会引入额外权限和兼容性问题。
- 每次继续打开同一 URL：不同浏览器对重复 URL 的复用策略不一致，无法保证不新建标签页。

## 5. 组件与职责

### 5.1 页面生命周期注册表

扩展现有服务端页面生命周期能力，使每个页面记录以下状态：

- 页面 ID。
- 最后心跳时间，用于现有十五分钟空闲退出策略。
- 最近活动顺序，用于稳定选择最近活跃页面。
- 当前事件连接及其刷新发送能力。

注册表提供独立的“请求刷新最近活跃页面”操作。它只在页面仍有有效事件连接时选择该页面，向唯一目标发送一次刷新指令，并返回是否成功找到目标。断开的页面不能成为刷新目标。

现有 `closePage`、过期清理和空闲退出语义保持不变。页面关闭、刷新或连接断开时移除对应连接；页面刷新完成后以新页面 ID 重新注册。

### 5.2 服务端生命周期接口

新增两类本地生命周期接口，均沿用当前 Bearer 会话令牌和可信 Origin 校验：

- 页面事件接口：浏览器页面使用带认证头的流式 `fetch` 建立事件连接，并携带自身页面 ID。
- 重开接口：macOS 启动器请求刷新最近活跃页面，服务返回是否已向某个页面发送刷新指令。

事件流只承载 CR 内部页面指令。本次仅定义 `reload` 指令，不扩展为通用消息系统。服务端在连接建立时发送初始事件或注释以确认连接已经就绪，连接关闭时立即注销。

### 5.3 浏览器页面

`PageHeartbeat` 继续负责活动心跳和关闭通知，并在挂载期间启动页面指令监听：

1. 页面生成稳定的本次加载页面 ID。
2. 发送首次心跳并建立事件连接。
3. 收到属于该页面的 `reload` 指令时，只执行一次 `window.location.reload()`。
4. 组件卸载或页面关闭时中止事件连接，并继续发送现有关闭通知。

事件连接短暂断开时，页面使用有限退避重新连接；重连不会刷新页面。认证失败等不可恢复错误不会形成高频重试，页面业务功能仍可继续使用。

### 5.4 macOS 启动器

`ServiceLauncher.launchOrReuse()` 根据服务状态分流：

- 新服务：保持现有健康检查和 `NSWorkspace.shared.open(state.launchURL)` 行为。
- 可复用服务：先调用本地服务的重开接口。
  - 返回已刷新：不打开 URL，只尽力激活负责该 HTTP URL 的系统默认浏览器应用。
  - 返回无可用页面：调用 `NSWorkspace.shared.open(state.launchURL)` 新开页面。
  - 请求超时、响应无效或接口失败：按“无可用页面”回退到打开 URL，避免 CR 点击后无响应。

默认浏览器激活与 URL 打开分离。启动器通过 macOS 对该 URL 的默认处理应用定位浏览器，并激活已经运行的应用实例；不写死浏览器 Bundle Identifier，也不切换用户默认浏览器。激活失败不撤销已经发送的刷新指令，也不额外打开 URL。

## 6. 数据流

### 6.1 首次启动或无现存页面

1. CR 启动或复用本地服务。
2. 启动器确认没有可刷新的页面，或刚刚启动了新服务。
3. 启动器把带一次性启动令牌的本地 URL 交给系统默认浏览器。
4. 页面加载、保存会话令牌、发送心跳并建立事件连接。

### 6.2 已有页面时再次点击 CR

1. AppKit 重开回调进入现有并发门。
2. 启动器验证 `service.json`、进程身份和健康状态。
3. 启动器携带服务会话令牌请求重开接口。
4. 服务从仍在线的页面中选择最近活跃页面，并只向该连接发送 `reload`。
5. 目标页面执行完整刷新；其他 CR 页面不处理该指令。
6. 启动器激活默认浏览器应用，但不打开 URL。

## 7. 竞争条件与错误处理

- 连续点击：继续使用 `OpenRequestGate`，同一时刻只允许一个重开请求。
- 页面在选择后立即关闭：事件连接关闭会注销目标；若刷新请求未能找到有效连接，则启动器新开页面。极窄的“指令已发送后页面关闭”竞争窗口不再额外开页，避免一次点击同时产生刷新和新标签。
- 多标签活动时间相同：使用服务端单调递增的活动顺序决定唯一目标。
- 页面刷新：旧连接和旧页面 ID 被清理，新页面加载后重新建立连接，不重复消费旧指令。
- 服务重启：内存中的页面注册自然清空；启动器按无页面路径打开新标签。
- 推送连接失败：页面有限退避重连；启动器无法找到有效连接时回退到打开 URL。
- 浏览器激活失败：保留已完成的页面刷新，不调用打开 URL，防止产生重复标签。
- 安全：所有新增接口只监听现有回环地址，继续校验会话令牌和可信 Origin；令牌不新增到事件 URL 查询参数或日志中。

## 8. 测试策略

严格按测试先行实施，覆盖以下行为：

### 8.1 服务端单元与路由测试

- 一个在线页面时，刷新请求只发送给该页面。
- 多个在线页面时，只发送给最近收到活动心跳的页面。
- 最近活动页面断开后，选择下一个仍在线的最近活跃页面。
- 没有在线页面时返回未刷新。
- 页面过期、关闭和服务空闲退出保持现有语义。
- 页面事件接口和重开接口拒绝无效令牌、非可信 Origin 与非法页面 ID。

### 8.2 Web 测试

- 页面挂载后发送心跳并建立指令监听。
- 收到 `reload` 时调用一次页面刷新。
- 非 `reload` 或格式错误的事件不刷新页面。
- 卸载时中止连接并保留关闭通知。
- 多个页面监听实例中，只有收到目标指令的实例刷新。

### 8.3 Swift 启动器测试

- 新服务就绪后打开默认浏览器 URL。
- 复用服务且刷新成功时不打开 URL，并尝试激活默认浏览器。
- 复用服务但没有页面时打开 URL。
- 重开请求失败或超时时打开 URL。
- 浏览器激活失败时不因回退再次打开 URL。
- 现有并发点击保护、服务进程身份校验和退出清理继续通过。

### 8.4 集成验收

1. 首次启动 CR App 后浏览器新增一个 CR 标签页。
2. 再次点击 CR App，标签页数量不变，原 CR 页面完成刷新。
3. 打开两个 CR 标签页，先后操作它们，再次点击 CR App 时只有最后操作的页面刷新。
4. 关闭全部 CR 标签页，再次点击 CR App 时新增一个 CR 标签页。
5. 停止本地服务后再次点击 CR App，服务重新启动并打开页面。
6. 执行全部 JS/TS、Swift、Playwright、类型检查、生产构建、仓库卫生、macOS smoke 和签名验证。

## 9. 范围外事项

本次不强制浏览器切换到具体 CR 标签页，不读取或修改浏览器历史，不申请辅助功能或 Apple Events 权限，不绑定具体浏览器，不改变 CR 页面 UI，不改变十五分钟空闲退出策略，也不引入面向远程网络的推送能力。
